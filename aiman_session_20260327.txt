  Yes, it's possible — and now implemented

  How it works

  When you launch a session from a JIRA issue, the system now:

  1. Stores the full JIRA issue (not just the key) in SessionConfig.Issue when
  selected in the issue picker
  2. Writes .aiman_task.md to the remote worktree with structured issue context:
  # PROJ-42: Implement user authentication

  **Status:** TODO
  **Assignee:** pieter

  ## Description
  We need OAuth2 support with Google and GitHub providers...
  3. Injects an initial prompt into the agent command so it starts working
  immediately:
    - Claude Code: claude --dangerously-skip-permissions "Read .aiman_task.md 
  for the JIRA issue details. Gather necessary codebase context and prepare an 
  implementation plan."
    - Gemini: Same pattern with gemini --yolo "..."
    - Cursor: cursor-agent --force . --message "..."
    - OpenCode: opencode-cli "..."

  Architecture

  The changes flow through three layers:
  - UI (dashboard.go): Stores issuePicker.selected as sessionCfg.Issue
  - UseCase (flow_manager.go): Passes the issue through to
  SkillEngine.PrepareSession (also fetches it during slugification if not
  already set)
  - Infrastructure (skills/engine.go): writeTaskFile() writes the markdown, each
   agent-specific preparer appends the positional arg/flag

  Key design decisions

  - File-based context (.aiman_task.md) avoids shell-escaping issues with
  complex JIRA descriptions
  - Short fixed initial prompt as a CLI arg just points the agent at the file —
  no special characters to worry about
  - Sessions created without a JIRA issue are unaffected (issue is nil, no task
  file written, no initial prompt)

⏺ All passing. Here's a summary of the safety fix:

  The problem: Passing a positional argument or --message flag to Gemini,
  Cursor, OpenCode, etc. could cause them to run the prompt headlessly and exit,
   killing the tmux session.

  The fix: PrepareSession now returns a PreparedSession struct with two fields:
  - Command — the agent launch command (never includes the prompt for unverified
   agents)
  - InitialPrompt — if non-empty, sent via tmux send-keys after a 3-second delay

  Per-agent behavior:

  ┌───────────┬─────────────────────────┬──────────────────────────────────┐
  │   Agent   │     Prompt delivery     │               Why                │
  ├───────────┼─────────────────────────┼──────────────────────────────────┤
  │ Claude    │ Inline positional arg   │ Confirmed: claude "msg" stays    │
  │ Code      │ in Command              │ interactive                      │
  ├───────────┼─────────────────────────┼──────────────────────────────────┤
  │ Gemini    │ tmux send-keys after 3s │ Positional arg behavior          │
  │           │                         │ unverified — could exit          │
  ├───────────┼─────────────────────────┼──────────────────────────────────┤
  │ Cursor    │ tmux send-keys after 3s │ --message behavior unverified —  │
  │           │                         │ could exit                       │
  ├───────────┼─────────────────────────┼──────────────────────────────────┤
  │ OpenCode  │ tmux send-keys after 3s │ Positional arg behavior          │
  │           │                         │ unverified — could exit          │
  └───────────┴─────────────────────────┴──────────────────────────────────┘

  The tmux send-keys approach is agent-agnostic — it just types the prompt into
  whatever interactive session is running, as if the user typed it themselves.
  The 3-second delay gives the agent time to fully initialize before receiving
  input. It runs as a background nohup process so it doesn't block session
  creation.
